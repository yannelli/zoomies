# Installing Zoomies

This document covers the two supported install paths for the Zoomies
control plane, plus the environment-variable reference and a first-run
checklist. Read [`OPERATIONS.md`](./OPERATIONS.md) for the NGINX
permissions / shared-group strategies the native install assumes.

## Prerequisites

- An NGINX install you control (Zoomies will write files into a directory
  you point at via `ZOOMIES_NGINX_SITES_DIR` and call `nginx -s reload`).
  Operator-managed NGINX is the v1 contract — the Docker Compose path
  ships an NGINX sidecar to satisfy this.
- One of: Node 22 LTS (for the native install) or Docker 24+ (for the
  Compose path).
- A domain pointing at the host's port 80/443 for production. `localhost`
  is fine for kicking the tires.

## Path A: Docker Compose

The shipped `docker-compose.yml` runs three services:

| Service  | Role                                                                              |
| -------- | --------------------------------------------------------------------------------- |
| `app`    | The Next.js control plane (UI + Route Handler API).                               |
| `worker` | The optional ACME renewal worker. Skip it if you only run `tls_mode: off/manual`. |
| `nginx`  | The edge proxy. Reads sites generated by `app` and serves traffic on port 80.     |

```bash
git clone https://github.com/yannelli/zoomies.git
cd zoomies
cp .env.example .env       # if you choose to maintain one; not shipped
# Edit .env: set ZOOMIES_API_TOKEN at minimum.
docker compose up -d
```

Required env (set in `.env` or an override file):

- `ZOOMIES_API_TOKEN` — bearer token for the HTTP API. Generate something
  long, treat it as a secret.
- `ZOOMIES_ACME_EMAIL` — only if you run the `worker`. The email
  registered with the ACME directory.

Optional env: see the reference table below. Defaults are wired into the
Compose file for the in-container paths (`/var/lib/zoomies` for state and
certs).

To run **without** the renewal worker, omit it from the up command:

```bash
docker compose up -d app nginx
```

To use the GHCR-hosted image instead of building from source, swap the
service `build:` block for `image: ghcr.io/yannelli/zoomies:latest` (or
pin to a specific tag).

The `app` container exposes port 3000 to the Compose network only; the
operator's `nginx` service is the public entrypoint on port 80.

## Path B: Ubuntu native (22.04 / 24.04)

This path uses the operator's existing NGINX and registers Zoomies as a
systemd service.

```bash
git clone https://github.com/yannelli/zoomies.git
cd zoomies
sudo ./scripts/install-ubuntu.sh
```

The script:

- Installs Node 22.x from NodeSource if `node` is not already on `$PATH`.
- Copies the working tree to `/opt/zoomies` and runs `pnpm install + build`.
- Registers `zoomies.service` (control plane) under
  `/etc/systemd/system/`.
- Drops a sample NGINX site at `/etc/nginx/sites-available/zoomies` and
  enables it.

Two systemd units are shipped:

- `zoomies.service` — the long-running control plane. Started by the
  installer.
- `zoomies-worker.service` — the ACME renewal worker. **Not** enabled by
  default. Enable it once you have populated `/etc/zoomies/worker.env`
  with at least `ZOOMIES_ACME_EMAIL` and `ZOOMIES_API_TOKEN`:

  ```bash
  sudo systemctl enable --now zoomies-worker.service
  ```

Set the env vars below in `/etc/systemd/system/zoomies.service.d/override.conf`
via `Environment=KEY=value` lines (or `EnvironmentFile=`) before enabling
the service in production. See `docs/OPERATIONS.md` for the
permission-sharing strategies (recommended: a shared `zoomies-mgmt` group
that owns the sites dir and is readable by both `zoomies` and `nginx`).

## Environment variable reference

| Variable                     | Default                                          | Required             | What it controls                                                                                                  |
| ---------------------------- | ------------------------------------------------ | -------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `ZOOMIES_API_TOKEN`          | _none_                                           | yes (production)     | Bearer token gating the HTTP API + UI cookie session. Without it, the API rejects every request.                  |
| `ZOOMIES_STATE_DIR`          | `<cwd>/.zoomies` (dev), `/var/lib/zoomies`       | recommended          | Holds the SQLite database, the ACME challenge directory, and worker bookkeeping. Must be writable by the process. |
| `ZOOMIES_NGINX_BIN`          | `/usr/sbin/nginx`                                | when not on PATH     | Path to the NGINX binary. Used for `-t` validation and `-s reload`.                                               |
| `ZOOMIES_NGINX_SITES_DIR`    | `/etc/zoomies/nginx/sites`                       | recommended          | Where Zoomies writes `<site>.conf` files. Must be inside an `include` in your `nginx.conf`.                       |
| `ZOOMIES_HEALTH_CHECK_URL`   | _none_                                           | for reload probes    | URL probed after each reload to confirm NGINX came back up. Typically `http://127.0.0.1/healthz`.                 |
| `ZOOMIES_ACME_EMAIL`         | _none_                                           | for `tls_mode: acme` | Contact email registered with the ACME directory.                                                                 |
| `ZOOMIES_ACME_DIRECTORY_URL` | `https://acme-v02.api.letsencrypt.org/directory` | no                   | Override the ACME directory — for example, point at the Let's Encrypt staging endpoint while testing.             |
| `ZOOMIES_CERT_DIR`           | `${ZOOMIES_STATE_DIR}/certs`                     | no                   | Where issued cert/key files land. Mounted into NGINX in the Compose path.                                         |
| `ZOOMIES_API_URL`            | `http://localhost:3000`                          | CLI only             | Default base URL the CLI hits in HTTP mode.                                                                       |

## First-run checklist

1. Hit `http://localhost:3000/login` (or your install's hostname) and
   submit the token configured in `ZOOMIES_API_TOKEN`. You should land on
   the dashboard.
2. Create an upstream — pick a name, choose the load-balancing policy,
   add one or more `host:port` targets with their weights.
3. Create a site — set the hostname, link the upstream, choose a
   `tls_mode` (`off` is fine for the first test).
4. SSH to the host and run `nginx -t` to confirm the rendered config is
   valid, then `nginx -s reload`. **In v1 the control plane does not
   auto-reload NGINX after a mutation** — you trigger the reload from the
   operator side (or hit `POST /api/v1/reload` from the CLI / API, which
   does run the validate-write-reload-probe cycle).
5. Curl the site's hostname and confirm traffic flows.

## CLI usage

The `zoomies` CLI ships as a binary in the npm package and as the
`dist/index.js` artifact in the runtime container.

- `zoomies --local …` — runs the command directly against the SQLite
  database, bypassing the HTTP API. Useful for one-off admin work on the
  same host as the control plane.
- `zoomies …` (default) — speaks to the HTTP API. Honours
  `--api-url <url>` (or `ZOOMIES_API_URL`) and `--api-token <token>` (or
  `ZOOMIES_API_TOKEN`).

`zoomies --help` lists the full command surface (sites, upstreams,
certs, reload, status).
